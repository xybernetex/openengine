# Training Guide

This document explains how the Xybernetex AI OpenEngine policy is trained,
how to extend it with new goal types, and how to fine-tune it after adding
new tools.

---

## Overview

Training happens in two phases:

1. **Behavioural Cloning (BC)** — supervised pre-training on synthetic
   demonstrations generated by `rollout_generator.py`.
2. **PPO Fine-tuning** — policy gradient training using the `ppo_trainer.py`
   loop and the `XybernetexEnv` gym environment.

The base checkpoint distributed with this repo was trained on ~50 k synthetic
demonstrations across the full goal taxonomy, then fine-tuned for ~1 M PPO
steps.

---

## Behavioural Cloning

### What it does

BC gives the policy a warm start.  Without it, PPO would spend enormous
compute exploring random action sequences before discovering even weak signal.

BC trains the policy by maximum-likelihood on (state, action) pairs from
offline demonstrations, treating it as a standard classification problem:

```
loss = CrossEntropy(policy_logits(state), expert_action)
```

### How demonstrations are generated

`training/rollout_generator.py` creates synthetic demonstrations by:

1. Sampling a goal from the **goal taxonomy** (see below).
2. Running a scripted "oracle" policy that always selects the correct action
   sequence for that goal class.
3. Recording (state vector, action index) pairs at each step.
4. Writing the episode to a `.jsonl` demonstration file.

The oracle policy is hard-coded per goal class using rules like:

```
PLAN → INVOKE_TOOL (×N) → ANALYZE → WRITE_ARTIFACT → DONE
```

This is intentionally simple.  The goal is not to record perfect behaviour but
to give the network a warm start that is directionally correct.

### Running BC

```bash
python training/rollout_generator.py --goals 10000 --output data/demos.jsonl
python training/ppo_trainer.py --mode bc --demos data/demos.jsonl --epochs 5
```

The BC checkpoint is saved to `models/bc_policy.pt`.

## PPO Training

### How it works

`training/ppo_trainer.py` wraps `engine/xyber_env.py` (an OpenAI-Gym
compatible environment) and trains with Proximal Policy Optimisation.

The training loop:

1. Roll out the current policy for `rollout_steps` steps across `n_envs`
   parallel environments.
2. Compute advantages using Generalised Advantage Estimation (GAE).
3. Run `ppo_epochs` gradient update passes over the collected data.
4. Log metrics (policy loss, value loss, entropy, mean reward) to stdout /
   Weights & Biases.
5. Save a checkpoint every `save_interval` updates.

### Running PPO

```bash
python training/ppo_trainer.py \
    --init-from models/bc_policy.pt \
    --rollout-steps 2048 \
    --n-envs 8 \
    --ppo-epochs 4 \
    --total-steps 1000000 \
    --output models/ppo_policy.pt
```

### Key hyperparameters

| Parameter | Default | Notes |
|---|---|---|
| `learning_rate` | 3e-4 | Cosine-decayed to 1e-5 |
| `gamma` | 0.99 | Discount factor |
| `gae_lambda` | 0.95 | GAE smoothing |
| `clip_epsilon` | 0.2 | PPO clip ratio |
| `entropy_coef` | 0.01 | Entropy bonus weight |
| `value_coef` | 0.5 | Value loss weight |
| `max_grad_norm` | 0.5 | Gradient clipping |

## Adding New Goal Types

The goal taxonomy determines what kinds of goals the policy is exposed to
during training.  It lives in `training/rollout_generator.py` as a dict of
goal class → generator function.

### Step-by-step

1. **Define a new goal class** in the taxonomy dict:

   ```python
   GOAL_TAXONOMY = {
       ...
       "data_analysis": generate_data_analysis_goal,
   }
   ```

2. **Write a generator function** that returns a dict with `goal_text`,
   `aspects`, and `oracle_actions`:

   ```python
   def generate_data_analysis_goal() -> dict:
       return {
           "goal_text": "Analyse the sales CSV and produce a summary report.",
           "aspects": ["load_data", "compute_stats", "write_report"],
           "oracle_actions": [
               "PLAN", "EXECUTE_CODE", "EXECUTE_CODE", "ANALYZE",
               "WRITE_ARTIFACT", "DONE"
           ],
       }
   ```

3. **Regenerate the BC demonstrations** with the new goal class included.

4. **Re-run BC** on the combined demonstration set.

5. **Fine-tune with PPO** starting from the new BC checkpoint.

## The Pluggable Reward Interface

The reward function is decoupled from the environment so you can define
domain-specific signals without touching the training loop.

### Using the default reward

```python
from training.reward import get_default_reward_function

reward_fn = get_default_reward_function()
```

### Writing a custom reward

```python
from training.reward import RewardFunction, TransitionData

class ResearchReward(RewardFunction):
    """Reward that additionally bonuses finding high-quality sources."""

    def compute(self, transition: dict) -> float:
        td = TransitionData(**transition)
        base = -0.1  # step penalty
        if td.exec_success:
            base += 0.3
        if td.done and td.aspects_after > 0:
            base += 15.0
        # domain bonus: check if extra dict has quality flag
        if td.extra.get("high_quality_source"):
            base += 1.0
        return base
```

Pass your reward function to the trainer:

```python
python training/ppo_trainer.py --reward-module myproject.rewards:ResearchReward
```

See `training/reward.py` for the full API, including `RewardConfig` for
adjusting individual scalar weights without subclassing.

## Retraining After Adding New Tools

When you add new tools to the ToolServer, the state vector changes because
the tool availability flags are updated.  This means the existing policy
checkpoint is no longer compatible with the new state dimension.

The recommended workflow is:

1. Add the new tool to the manifest and register it with the ToolRegistry.
2. Update `STATE_DIM` in `engine/rl_chassis.py` if the tool count changed.
3. Re-generate BC demonstrations that include episodes using the new tool.
4. Run BC from scratch (or from the old checkpoint with the new input layer
   re-initialised).
5. Fine-tune with PPO.

If the new tool is purely additive (i.e., you are not changing existing tool
semantics), you can often skip full BC and do a short PPO fine-tune from the
existing checkpoint after re-initialising just the input projection layer.

## Goal Taxonomy Design Tips

- **Cover the full action space.** Make sure your taxonomy includes goals
  that require every primitive action.  A taxonomy without any code-execution
  goals will produce a policy that never takes `EXECUTE_CODE`.
- **Vary episode length.** Include short goals (3–5 steps) and longer goals
  (10–15 steps) so the policy learns to use the step budget efficiently.
- **Balance goal classes.** Heavily imbalanced taxonomies cause the policy to
  specialise.  Aim for rough parity in the number of demonstrations per class.
- **Include failure demonstrations.** A small fraction (5–10 %) of
  demonstrations that reach max steps without completing the goal teaches the
  policy to recognise the stagnation penalty and avoid it.
- **Aspect granularity matters.** Too few aspects (1–2) provides weak shaped
  reward.  Too many (10+) dilutes each completion bonus.  3–5 aspects per goal
  is a good target.
